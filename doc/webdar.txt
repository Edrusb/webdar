WEBDAR PROJECT

Introduction:
       The following is a description of software project from the
       services it should provide to the possible way it could be
       implemented. This is a proposal, subject to dscussion in all
       its aspects.

AIM:
        The main targets for the initial release are:
        - provide a web-base interface to libdar
        - it should be usable with simple browser like lynx or w3m
        - allow remote operation through web interface
        - let libdar store slices to remote storage through FTP as an
	  alternative to local filesystem.
        - propose a basic authentication mecanism (for now) like
          dynamically generated password displayed on stdout for use
          throught the web session.
	- for the initial release, only few Unixes systems will be
	  supported but at least Linux and MacOS X.


Form and Constraints:
        - webdar must be available as a single executable
          suitable for use in bootable Live-CD systems.
          It may be available as shared and statically linked
          binary.
	- It must not rely on any existing directory tree to provide
 	  a web service (a single static binary should work alone)
	- It shall be able to work even on a read-only filesystem,
	  and should not need to pass through filesystem to provide
	  a gateway between libdar and http protocol (this would cause
	  performance pernalty during backup as most of the disk bandwidth
	  would be used by the backup/restoration process).
	- it shall not rely on any non GPL or proprietary software, both
	  for building the software and for working.



LICENSING:
	Webdar will be released under GPLv3. Version 3 is important
	for Webdar as it could be easily emmbedded in many proprietary
	devices.

POSSIBLE EVOLUTIONS:
        Room should be done for future evolutions:
         - support for SSL support between browser and webdar
         - support for other network protocols than FTP (to be defined)
         - support for daemon mode and user priviledge management
         - support for concurrent sessions (may not be available in version 1.0)


IMPLEMENTATION DISCUSSION:

- Threads support:
	It should be good to have a multi-threaded application from the
	beginning, this will ease further support of concurrent sessions and
	daemon monde. Moreover, this will let webdar be able to report operation
	progression during backup/restoration (libdar provides such a mecanism
	that was requested by kdar some years ago).

- Network awareness:
	FTP modules (FTP for initial release) must not be part of libdar to not
	bring unnecessary dependencies and security risks to existing tools
	relying on libdar. It might be possible in a future release to split
	the network part from webdar into an autonomous library independant
	of libdar (named libwebdar for example) if some other GPL software
	need to use it.

- Language:
	libdar's API is provided in C++. While C++ is probably the most simple
	way to implement webdar, nothing obliges the use of a particular
	language to implement Webdar. However, need will be to provide binding
	to libdar if C++ is not retained for the implementation. Note also that
	libdar strongly uses classes an objects, choosing an object oriented
	language would cause less work for the binding and no work at all if C++
	would be chosen.

PHASING:
	DESIGN:
	a first phase is to describe the classes their interaction and from this
	to formalize the classes definitions. This must be done first before any
	implementation starts not to loose time developping one way and restarting
	from scratch another way, becauses class specs changed in the meanwhile.

	IMPLEMENTATION
	Once all classes role is well defined, and once the classes definition
	(their public part) is documented in the way they are expected to be
	used, implementation shall start. Each class should be under the
	responsibility of a developper, which has all freedom to define the
	 implementation.

	It is probable (but not wanted) that some aspects related to the
	design phase shows during the implementation phase. Either this could
 	be minor change, that shall be done and communicate to all other
	developpers for agreement, or this is could be a major change, that
	reveals a major	problem in the design. In that case all development
	should stop as soon as possible and a review of the design shall become
	the major focus for all team members.

        TESTING PHASE
	For non trivial classes, or set of classes, it should be a good point to
	implement some sample programs using them these classes testing. Such
	tools should be provided beside webdar source code for testing in future
	releases and maybe adapted as non-regression tools

	Then a global testing could take place. Testing against different
	browsers, different FTP servers, different OS.

	RELEASE PHASE
	Probably a pre-release phase should be interesting to get progressive
	audience and feedback on webdar, fix bugs in order to start with an
	initial official release enough stable to not have webdar become ruined
	by a bad reputation of buggy software.


DESIGN CONSIDERATION:

THREADS
        main thread:
	 when starting webdar, the process, set signal handlers, and fires new
	 thread if more than one port is to be listen to, or launches a listener
	 object in each thread. All listener's end leads to the program end.
	    An alarm or SIGINT signal may be set to allow clean stop of the
	    daemon (no more listening, waiting for all threads to end before
	    exiting the program).
	- holds a list of current running operation
	- listen on TCP port
	- upon connexion
	   - does the authentication
	   - if the current thread list is not empty
	     - show the list of running thread (future: for the authenticated user)
	     - let the user select the thread and pass the hand to the selected session
	   - if the current thread-list is empty
	     - creates a new thread
	     - and pass the session control to the thread

        The list of current threads is a list of objects of class job
	    the job class holds the thread id, status (connected, notconnected),
	    job running / not running
	    it must use mutex to be updated by the thread when it is disconnected
	    when it is finished before the thread ends,


	Each job thread may be either in configuration mode, or in running-mode.
	In running mode, a child thread is running the libdar call for the requested
	operation. The job thread only displays the report on periodic manner. It also
	allows the user to ask for interruption which can be transmitted to child thread
	thanks to libdar mechanism.

URL
	The site specific URL, (the part after http://hostname/) must first
	be related to the session.

	as described later, the generation of an html page is done by a hierarchy of objects.
	The rest of the URL follows this structure.

	the thread is statefull and does modifies orders from page to page with an increment
	number to avoid browser'back and forth to apply twice the same action. Former orders
	are ignored and are responded by the current contents status.

	the structure of URL should be:

	    	{protocol}://<hostname>/<thread ID>/<tag>/<tag>/.../<tag>/

	at startup, the URL

		{protocol}://<hostname>/

	will result
	 - in authentication request,
	 - cookie assignment,
	 - for the authenticated user, either the list of existing session it owns or
	   directly a new session the output of a new or existing thread

	It must be avoided caching of generated pages, thus the no-cache directive must be
	set in replies, except for static contents:

	Some static contents like pictures may be refered in pages as

		{protocol}://<hostname>/static/<object>

        for this contents, the caching by browsers is allowed, there is not required cookie for
	that contents.

	There is no obligation that a TCP session receive requests for the same session.

	if the receiving thread is a static thread (one providing static contents only), the
	process is the same except that upon success the former thread ends.

	If request for a target session does not have the correct cookie, an error message
	is reported (missing cookie) and lead to the main page (authentication, etc.).


AUTHENTICATION

	when the authentication should be performed :
	- in absence of cookie
	- if the requested session does not exist
	- if the cookie is incorrect for the requested session

CLASSES

	thread : pure virtual class which objects may run in a particular thread. class listener and class
	    session depends on this class

	req_ans : pure virtual class that defines the interface of answers and requests
	(implemented by sessions and chooser objects)

	connexion : this class is a wrapper around socket filedescriptor to provide generic_file interface

	session : this class provides objects that store the status of a particular session, like options selected
	    by the user current configuration status. This information is stored as tree or objects generating HTML code
	    for which the session object is the root. When comes the time to call libdar, the object owns a thread
	    that updates object's structure to be able to have progession information as well as being able to abort
	    the operation in progress. session class is an implementation of req_ans.

	    Session object management is done by the class itslef. The class provides a table of existing sessions,
	    with their associated properties.

	    A mutex is used for each object to avoid a session to be solicited for several requests at the same time.
	    Prior to answer a request, the client must provide the appropriated cookie, the one used at session creation
	    or obtained by a chooser object.

	parser : (inherit from thread) this class provides threaded objects that parse requests from given connexion, only
	    the HTTP protocol aspect is managed here, no HTML genereation is done heer. The class holds a counter
	    on the number of running parser (=connexions), if a maximum has been defined the constructor fails if the
	    counter has reached this threshold.
	    The parser object owns the connexion that it has been given by the listener object that it has been createed by,
	    and when the connexion ends the parser thread also ends, the parser object is also destroyed.
	    The parser, lookup the session object based on the requested URL
	    and if found transmit the request to the found object. The answer provided by the session object is returned
	    to the client. An parser may send request to different sessions, depending
	    on the nature of the requests received from the connexion.

	    Some static content (like images, logo, or documentation) may be provided directly by the parser from a
	    catalog of contents.

	    A parser object may receive a request that does not target a particular session. In that case the parser	authenticates
	    the client and upon success, calls a chooser object instead of a session object.


	listener : (inherit from thread) a listener object is bound to a port and eventually to an IP address,
            this is a threaded object
	    Its role is to wait for incoming connexions and to create connexions objects and a parser object,
	    giving it the connexion
	    the parser object being a threaded object manages all further actions independly of his listener parent
	    or from other listener. it may in future release use a derivated "connexion" class that provide
	    SSL encryption/decryption and authentification. The listener class provide a mean to signal the end
	    request to all its members (used by signal handler).

	chooser : chooser is an implementation of req_ans
	    a chooser object does a lookup of the session table. If no session object owned by the authenticated user exist,
	    it manages for a new one to be created
	    and transmit the request to this newly object and send back the answer to it its caller (a parser). Then it dies, further
	    exchanges are done directly between the parser and the session object.
	    if several session owned by the authenticated user exists, the chooser provides a list of them as answer to the request
	    and dies.

LIBDAR CLASSES

	web_interaction : inherits from libdar's user_interaction
 		the role is to implement the user interaction requested by libdar either for asking questions
		to the user or to provide reporting on the operation under progress

	due to the nature of HTTP, where no mean are available to initiate a transaction by the
	server, this class has the role to buffer output pending for to built the next HTTP answer
	and buffer pending requests to provide a mean to the user to answer it after the next request he will do.

	Second point is the nature of the threads used. One tread is running libdar, while a second is
	managing user interaction by mean of HTTP protocol (parser object/thread solliciating a session object)
	First, no HTTP session may be active for a particular libdar job, leading the job to be suspended
	if libdar need user's decision. Second even if a parser thread is actively associated with this libdar
	job, the libdar thread must be suspended up the next HTTP request, the answer with libdar question in it
	and the new request with the user's answer, (then the aswer with pre-libdar interaction output).

	This is tiplically a mutex used as semaphore is necessary (a mutex not protecting a critical session,
	but "up" by default, on which the libdar thread will be pending, and set to down by the HTTP thread, to
	release the libdar thread when the answer will be provided).
